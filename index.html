Project Overview
This pipeline automates the retrieval, scanning, and compliance reporting of political ad creatives from Freewheel before they go live. It connects Freewheel (ad source) → IronDome (spec scanner) → ComplyAI (compliance team) → Ad Ops (final approve/reject).

Objective
Build an automated, hourly pipeline that retrieves political ad creatives from Freewheel before they go live, validates them through IronDome (internal creative scanner), and delivers structured compliance-ready outputs to the Comply AI team in their required JSON schema.

Today, political creatives entering the Freewheel system are not systematically reviewed before going live. There is no automated mechanism to detect, extract, or flag them for compliance review — creating regulatory exposure and operational risk during political campaign seasons. This pipeline makes that process automated, auditable, and repeatable.

Technical Implementation
Architecture, data flow, and stage-by-stage implementation guide. Python-based scheduled service with four independently-retriable stages. Draft v1.0
Architecture overview
#StageResponsibility
1Freewheel FetcherPoll Freewheel API, apply political industry filter, delta-cursor deduplication, pagination
2VAST ParserFetch and parse vast_content_url XML per creative — extract adDescription, fileFormat, fileSize
3IronDome ScannerSubmit each creative to IronDome for spec validation, collect scan results, build flagged list
4ComplyAI Formatter & SenderMap all fields to ComplyAI JSON schema, validate required fields, deliver two payloads with retry logic
Technology stack
Python 3.11+  ·  httpx / requests (HTTP client)  ·  schedule (cron)  ·  lxml (VAST XML parsing)  ·  pydantic (schema validation)  ·  boto3 (secrets manager + optional S3 delivery)  ·  structlog (structured logging)
Stage 1 — Freewheel Fetcher
Endpoint
GET /v1/creative/programmatic_creatives ?approval_status=PENDING &page=1 &per_page=100 &updated_since={LAST_RUN_UNIX_TS} Authorization: Bearer {FREEWHEEL_API_TOKEN} Accept: application/json
Political detection logic
def is_political(creative: dict) -> bool: return any( 'politic' in (ind.get('name') or '').lower() for ind in creative.get('industries', []) ) # Covers: 'Politicians - All else', 'Political Advertising', # 'Political - Federal', 'Political - State', etc.
Delta cursor state file
{ "last_run_ts": 1748300400, "last_run_iso": "2026-05-26T13:00:00Z", "last_run_status": "success" }
Stage 2 — VAST XML Parser
Three ComplyAI required fields are only available from VAST XML: adDescription (adTitle), fileFormat, and fileSize. A secondary HTTP fetch is required per creative.
def parse_vast(vast_url: str) -> dict: resp = httpx.get(vast_url, timeout=10) tree = etree.fromstring(resp.content) ad_title = tree.findtext('.//AdTitle', default='') media_files = tree.findall('.//MediaFile') file_format, file_size = '', 0 if media_files: mf = media_files[0] file_format = mf.get('type', '').split('/')[-1].upper() file_size = round(int(mf.get('fileSize', 0)) / (1024**2), 2) return { 'adDescription': ad_title, 'fileFormat': file_format, 'fileSize': file_size, }
Stage 3 — IronDome Integration
The design below assumes a synchronous REST API. If IronDome is async (webhook callback), Stage 3 will be split into a submission step and a polling/callback step.
def scan_creative(creative_id: str, vast_url: str) -> dict: resp = requests.post( url=IRONDOME_SCAN_ENDPOINT, headers={'Authorization': f'Bearer {IRONDOME_TOKEN}'}, json={'creativeId': creative_id, 'vastUrl': vast_url}, timeout=30, ) result = resp.json() return { 'passed': result.get('status') == 'PASS', 'notes': result.get('feedback', ''), }
Creatives where passed=False are collected into a separate flagged_creatives list. IronDome feedback string is written to the GeneralNotes field of the ComplyAI payload. If IronDome is unreachable after 3 retries, scan_status is set to UNKNOWN and the creative is included in the full list only.

Security
  • All API tokens (Freewheel, IronDome, ComplyAI) stored in AWS Secrets Manager — never in code, environment files, or version control
  • VAST XML fetched over HTTPS only; CloudFront origin URL validated against expected domain pattern
  • All inter-service calls use short-lived tokens refreshed at run start
  • Dead-letter queue payloads encrypted at rest
  • Run logs redact any PII or sensitive advertiser data before writing to log sink
